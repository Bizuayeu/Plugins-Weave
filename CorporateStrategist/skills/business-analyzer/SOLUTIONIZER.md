### 2.1 問題構造の特定（ボトルネック分析）

#### 目的

業務フロー・組織・市場におけるボトルネック（制約条件）を特定する

---

#### 分析観点

| 観点 | 特定対象 |
|------|---------|
| **業務フロー** | 処理時間が長い工程、手待ちが発生する箇所、手戻りが多い工程 |
| **組織構造** | 意思決定の遅延、情報の滞留、権限の不明確さ |
| **市場構造** | 参入障壁、規制、リソース不足 |
| **技術構造** | レガシーシステム、スキル不足、ツール欠如 |

---

#### ボトルネック特定の手法

**1. 制約理論（TOC: Theory of Constraints）の適用**

システム全体のスループットを制限している最も制約となっている要素を特定

```
Step 1: 制約の特定
    - 最も時間がかかる工程はどこか？
    - 最もリソースが不足している箇所は？

Step 2: 制約の徹底活用
    - その制約を最大限活用するには？

Step 3: 他をすべて制約に従属させる
    - 制約以外の工程は制約に合わせて調整

Step 4: 制約の能力を向上
    - 制約を解消する施策は？

Step 5: 新たな制約を特定
    - 制約が移動したら、再度Step 1へ
```

**2. 時系列分析**

業務プロセスの各工程にかかる時間を可視化

```
工程A（30分）→ 工程B（120分）→ 工程C（15分）
                    ↑
                ボトルネック
```

**3. 依存関係マッピング**

クリティカルパス（最長経路）を特定

```
タスクA（2日）┐
              ├→ タスクC（3日）→ タスクE（1日）
タスクB（1日）┘

クリティカルパス: A → C → E（6日）
ボトルネック: タスクC（3日）
```

---

#### アウトプット形式

```markdown
## ボトルネック分析結果

### ボトルネック1: [名称]

**分類**: 業務フロー / 組織構造 / 市場構造 / 技術構造

**影響**:
- 週/月あたりの時間消費: [X時間]
- 業務効率への影響: [Y%低下]
- 売上・利益への影響: [定量/定性評価]

**原因**:
- [根本原因1]
- [根本原因2]

**改善余地**:
- 大/中/小
- [理由]

**推奨施策**:
- [短期施策]
- [中長期施策]
```

### 2.2 暗黙知の可視化

#### 目的

属人化している知識・ノウハウを特定し、形式知化の可能性を評価する

---

#### 暗黙知の4類型

| 類型 | 説明 | 例 |
|------|------|-----|
| **経験知** | 長年の経験から得られた直感・判断基準 | 「この顧客は値引き交渉に応じやすい」 |
| **手続き知** | 手順・プロセスに関する知識 | 「この書類は先に総務に回す」 |
| **関係知** | 人間関係・ネットワークに関する知識 | 「この件は〇〇さんに相談すれば早い」 |
| **判断知** | 意思決定の基準・優先順位 | 「この条件なら受注すべき」 |

---

#### 暗黙知の特定手法

**1. 属人化マッピング**

特定の人にしかできない業務を洗い出し

```
質問例:
- 「〇〇さんが休むと困る業務は何ですか？」
- 「新人が一人でできない業務は何ですか？」
- 「マニュアル化されていない業務は何ですか？」
```

**2. リスク評価**

その人が離脱した場合の影響度を評価

| リスクレベル | 基準 |
|-------------|------|
| **Critical** | 業務が完全停止、代替者なし、習得に6ヶ月以上 |
| **High** | 業務に重大な支障、代替者1-2名、習得に3-6ヶ月 |
| **Medium** | 業務に支障、代替者複数、習得に1-3ヶ月 |
| **Low** | 軽微な影響、代替者多数、習得に1ヶ月未満 |

**3. 形式知化可能性の判定**

言語化・構造化できるかを評価

---

#### 暗黙知の形式知化可能性評価

| 評価 | 基準 | 対応 |
|------|------|------|
| **高** | 明確なルール・判断基準が存在、データ・ドキュメント有 | スキル化・マニュアル化を推奨（Phase 3へ） |
| **中** | ある程度言語化可能、専門家の協力が得られる | スキル化を検討、一部は人間判断 |
| **低** | 完全に感覚的、文脈依存度が極めて高い | 人間に依存、ペアワークで伝承 |

---

#### アウトプット形式

```markdown
## 暗黙知リスト

### 暗黙知1: [名称]

**保有者**: [担当者名]

**類型**: 経験知 / 手続き知 / 関係知 / 判断知

**内容**:
[暗黙知の具体的内容]

**属人化リスク**: Critical / High / Medium / Low
- [リスク評価の理由]

**形式知化可能性**: 高 / 中 / 低
- [評価の理由]

**推奨対応**:
- [スキル化 / マニュアル化 / ペアワーク伝承]
```

### 2.3 改善機会の抽出

#### 目的

ボトルネック・暗黙知の分析から、具体的な改善施策を提案する

---

#### 改善の3つの方向性

**1. 効率化**
- 業務時間の削減
- 自動化・スキル化による省力化
- プロセスの簡素化

**2. リスク低減**
- 属人化の解消
- ミス・法的リスクの軽減
- 知識の組織資産化

**3. 競争力強化**
- 意思決定の高速化
- サービス品質の向上
- イノベーション促進

---

#### 改善施策の優先順位付け（簡易版）

Phase 3で詳細に評価しますが、Phase 2の段階で以下の観点で優先順位を付けます：

| 観点 | 評価基準 |
|------|---------|
| **インパクト** | 業務効率化・売上貢献・リスク低減の効果（大/中/小） |
| **実現可能性** | データ整備状況・専門家協力・技術難易度（高/中/低） |
| **緊急性** | 市場環境・組織状況による実施時期の重要性（高/中/低） |

---

```markdown

- 時間削減: [週/月あたりX時間]
```

## Phase 3: Skill Architecture Design

**Phase 2で特定した暗黙知・ボトルネック業務のスキル化を設計**

---

### 3.1 スキル化対象の特定と優先順位付け

#### スキル化候補の特定基準

**✅ スキル化に適した暗黙知**:

1. **反復性が高い**
   - 週次・月次で頻繁に発生する業務
   - 同じパターンが繰り返し現れる判断
   - 例: 見積もり作成、リスク評価、レポート生成

2. **専門性が高い**
   - 習得に時間がかかる知識・経験
   - 特定の人しか持っていないノウハウ
   - 例: 技術仕様書レビュー、契約書チェック、財務分析

3. **属人化リスク大**
   - 特定の人に依存している業務
   - その人が離脱すると業務が止まる
   - 暗黙知が言語化されていない

4. **形式知化可能**
   - 言語化・構造化できる知識
   - 判断基準が明確化できる
   - データ・ドキュメントが存在する

**❌ スキル化に不適**:

- 完全に感覚的（言語化不可能）
- 文脈依存度が極めて高い
- 一回性の業務
- リアルタイム物理作業

---

#### 優先度スコアリング方式

**計算式**:
```
優先度スコア = インパクト(1-10) × 実装容易性(1-10) × 緊急度(1-10) / 1000
```

---

**【インパクト】(1-10)**

業務効率化・売上貢献・リスク低減の総合評価

| スコア | 基準 |
|--------|------|
| 9-10 | **Critical** - 年間500時間以上削減 / 売上10%以上貢献 / 重大リスク回避 |
| 7-8 | **High** - 年間200-500時間削減 / 売上5-10%貢献 / 中程度リスク低減 |
| 4-6 | **Medium** - 年間50-200時間削減 / 売上1-5%貢献 / 小リスク低減 |
| 1-3 | **Low** - 年間50時間未満削減 / 売上1%未満貢献 / リスク影響小 |

**評価要素**:
- 業務効率化: 何時間/週の削減が期待できるか？
- 売上貢献: 収益への直接影響度は？
- リスク低減: 属人化・ミス・法的リスクの深刻度は？

---

**【実装容易性】(1-10)**

データ整備・専門家協力・技術難易度の総合評価

| スコア | 基準 |
|--------|------|
| 9-10 | **Very Easy** - 完全なドキュメント有 / 専門家フル協力 / シンプルなロジック |
| 7-8 | **Easy** - 基本ドキュメント有 / 専門家協力的 / 標準的な処理 |
| 4-6 | **Medium** - 部分的ドキュメント / 専門家協力限定的 / 中程度の複雑性 |
| 1-3 | **Hard** - ドキュメント不足 / 専門家非協力的 / 高度な技術要求 |

**評価要素**:
- データ整備状況: 既存ドキュメント・手順書の有無
- 専門家availability: 知識保持者の協力可能性
- 技術的難易度: AI処理の複雑度

---

**【緊急度】(1-10)**

市場環境・組織状況に基づく実装時期の重要性

| スコア | 基準 |
|--------|------|
| 9-10 | **Urgent** - 3ヶ月以内必須（法規制・退職リスク・競合圧力） |
| 7-8 | **High** - 6ヶ月以内推奨（市場変化・業務逼迫） |
| 4-6 | **Medium** - 1年以内実施（中期戦略・段階的改善） |
| 1-3 | **Low** - 1年超でも可（長期的最適化） |

**評価要素**:
- 市場環境: 競合状況、法規制対応の緊急性
- 組織状況: 退職リスク、業務逼迫度

---

**【スコアリング例】**

| スキル候補 | インパクト | 実装容易性 | 緊急度 | **優先度スコア** |
|-----------|-----------|-----------|--------|-----------------|
| 契約書リーガルチェック | 9 | 8 | 8 | **5.76** |
| 見積書作成支援 | 7 | 9 | 6 | **3.78** |
| 財務分析レポート | 6 | 7 | 5 | **2.10** |
| 業界調査まとめ | 5 | 8 | 4 | **1.60** |

**優先順位判定**:
- **5.0以上**: 最優先（即座に着手）
- **3.0-5.0**: 高優先（3ヶ月以内）
- **2.0-3.0**: 中優先（6ヶ月以内）
- **2.0未満**: 低優先（1年以内）

### 3.3 運用戦略の策定

#### 運用設計の5つの観点

**1. トリガー設計**

**いつ・誰が・どのようにスキルを起動するか**

| 項目 | 設計内容 |
|------|---------|
| **起動タイミング** | 定期実行 / イベント駆動 / オンデマンド |
| **起動者** | 特定の役職 / チーム全体 / システム自動 |
| **起動方法** | Claudeへの明示的指示 / Slash Command / MCP統合 |

**設計例**:
```
トリガー: 見積もり依頼メールを受信した時
起動者: 営業担当者
起動方法: Claude Code で `/estimate` コマンド実行
```

---

**2. 権限設計**

### Phase 2 → Phase 3 判定基準

**Phase 3（スキル化設計）へ進むべき対象**:

以下の条件をすべて満たす場合、Phase 3へ進む：

1. **形式知化可能性: 高または中**
   - 言語化・構造化できる知識である

2. **スキル化適性: 4つの基準のうち3つ以上を満たす**
   - 反復性が高い
   - 専門性が高い
   - 属人化リスク大
   - 形式知化可能

3. **優先度: 高または中**
   - インパクト・実現可能性・緊急性の総合評価で一定以上

**Phase 2で完結すべき対象**:

- 形式知化可能性が低い（感覚的・文脈依存）
- 一回性の業務
- マニュアル化で十分対応可能
- スキル化のコストが効果を上回る
